Agent framework comparison

Which stack should you reach for?

Framework Best when Why teams pick it Watch-outs
OpenAI Agents SDK You want a clean code-first agent runtime plus tight access to OpenAI tools and evals. Fast path to tool-using agents, strong docs, sandbox agents, good fit for single and multi-agent orchestration. Best experience is in the OpenAI ecosystem; bring your own observability conventions if you want vendor-neutral tracing.
LangGraph You need durable, stateful, graph-shaped control over long-running workflows. Excellent for explicit orchestration, human-in-the-loop control, resumability, and agent state modeling. Lower-level than batteries-included frameworks, so you need to be comfortable designing the control flow yourself.
Google ADK You want workflow agents, memory, callbacks, and evaluation in a structured SDK. Clear primitives for tools, sessions, memory, and orchestration; strong Google ecosystem integration. Best fit if you are comfortable with the ADK mental model and its service abstractions.
Microsoft Agent Framework You want enterprise-oriented multi-agent workflows with type-safe orchestration. Strong for graph workflows, session state, middleware, and interoperability in large enterprise setups. Heavier than lightweight SDKs; public preview roots matter for change velocity even though 1.0 is production-ready.
DSPy You care about optimizing prompts and modules systematically, not hand-tuning prompt strings. Great for composing and compiling AI programs, especially when you want optimization loops for RAG or agent components. It is not primarily an operations harness; pair it with stronger runtime and tracing layers for production agents.
PydanticAI You want strong typed I/O, validation, and Python-native ergonomics for production-grade agents. Very good fit when structured outputs, correctness, and clean Python interfaces matter more than graph complexity. Use it when type safety is central; add separate workflow orchestration if you need deep multi-agent graph control.

Historical note: AutoGen mattered a lot, but its official repo is now in maintenance mode, so new builds should usually start with Microsoft Agent Framework or another actively evolving stack.

RAG pattern comparison

How advanced RAG patterns differ

Pattern What it adds Use it when Main tradeoff
Contextual Retrieval Prepend chunk-specific context before indexing to make chunk retrieval less ambiguous. Your data is fragmented and plain chunk embeddings miss document-level meaning. Higher indexing cost and preprocessing complexity.
Hybrid dense + sparse retrieval Combines semantic recall with lexical precision. You need better robustness across keyword-heavy, acronym-heavy, or exact-match queries. More knobs to tune around fusion and ranking.
Reranking Improves final relevance after broad retrieval. Base retrieval is decent but top results still contain noise. Extra latency and model cost.
Self-RAG The model decides when to retrieve and critiques its own generations. You want adaptive retrieval instead of always-on retrieval. More complex inference behavior and usually stronger modeling requirements.
CRAG Evaluates retrieval quality and triggers corrective actions when retrieval is weak. You want a plug-in robustness layer for imperfect corpora. Longer pipeline and more moving parts.
GraphRAG Uses structured graph abstractions for multi-hop or thematic discovery over complex corpora. Your questions need relational reasoning across many entities or themes. Indexing can be expensive and prompt tuning matters a lot.
HippoRAG Treats retrieval more like long-term associative memory using graph structure plus Personalized PageRank. You care about memory-like retrieval behavior and associative reasoning. More research-heavy than standard enterprise stacks.
LightRAG Targets simpler, faster graph-style RAG with a practical codebase. You want graph-enhanced retrieval without the full GraphRAG overhead profile. Still depends on good extraction quality and careful dataset fit.
Evaluation-first RAG Designs the retrieval system around continuous measurement, not just architecture novelty. You are moving from prototype to production. Requires datasets, metrics, traces, and iteration discipline.

What to Build Next

If you want to turn the reading list into skill, build a small but real system in each category.

Project 1

Retrieval-grounded research assistant

  • Use hybrid retrieval plus reranking.
  • Add answer citations and retrieval traces.
  • Evaluate recall, precision, faithfulness, and latency before tuning prompts.
Project 2

Tool-using operational copilot

  • Give the agent a small, well-described toolset and explicit approval boundaries.
  • Trace tool calls and grade trajectories, not just final answers.
  • Use a harness with retries, checkpoints, and clear failure surfaces.
Project 3

Memory-backed personal analyst

  • Store user preferences and durable facts outside the prompt.
  • Retrieve only context relevant to the current task.
  • Measure memory hit quality and rejection behavior on no-answer cases.
Project 4

Graph-augmented enterprise QA

  • Start with standard RAG, then add GraphRAG or HippoRAG only where baseline retrieval breaks.
  • Benchmark complex multi-hop questions separately from simple factual retrieval.
  • Track token cost and index-build cost alongside accuracy.

This atlas is intentionally opinionated: it prefers resources that help you build and debug real systems over resources that only explain the idea at a high level.
